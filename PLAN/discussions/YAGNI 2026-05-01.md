# Décisions YAGNI — 2026-05-01

> Quatre features livrées sont retirées du périmètre après usage réel. La règle YAGNI prime
> sur l'attachement aux specs : si personne ne s'en sert, on retire le code, les tests, la
> doc, les références dans le PLAN. La dette de portage future en sera réduite d'autant.

## 1. Heat map du débat (PHASE-19) — retrait complet

**Raison :** personne ne s'en sert. La géométrie du débat passe déjà par le dashboard de
consensus, la palette Wong, l'alignement et les pastilles. La couche thermique en plus
n'apporte pas de valeur observable.

**Impact code (à faire dans une session dédiée) :**
- `front/utils.py` : retirer le calcul du score d'intensité + injection CSS heat map
- `front/views.py` : retirer le flag `heatmap` du `LectureViewSet`
- `front/static/front/js/marginalia.js` : retirer le toggle heat map
- `front/static/front/css/` : retirer les variables `--heatmap-tiede`, `--heatmap-chaud`, `--heatmap-brulant`, classe `.heatmap-active`
- `front/static/front/js/keyboard.js` : retirer le raccourci `H`
- Templates : retirer le bouton de toggle heat map de la toolbar
- `front/tests/test_phase19*.py` : suppression
- Modale d'aide raccourcis : retirer la ligne `H`

**Impact PLAN :**
- INDEX.md : phase marquée DEPRECATED 2026-05-01
- PLAN.md : étape 1.15 à annoter DEPRECATED
- INSPIRATION_ATOMIC.md : sous-phase 37d (heat map sémantique) devient sans objet
- Graphe de dépendances : retrait de la branche `PHASE-09 → PHASE-19`

## 2. Mode lecture / focus (PHASE-17, partiel) — délégué à Firefox Reader View

**Décision :** retirer le mode focus tel qu'implémenté côté Hypostasia. Conserver les
autres raccourcis clavier (T, E, J/K, C, S, X, A, ?, Escape).

**Raison :** Firefox Reader View (Mozilla Readability.js) couvre déjà l'usage. La zone de
lecture d'Hypostasia est compatible (vérifié 2026-05-01) :
- `<h1>` clair pour le titre, `<article id="readability-content">`, classes `prose prose-slate prose-lg`
- Contenu en paragraphes avec `prose-p:mb-5`, ratio texte/HTML élevé
- Toolbar et overlays sont en `position: fixed` → ignorés par l'algorithme Readability
- L'utilisateur appuie sur F9 (Firefox) pour basculer en Reader View native

**Impact code (à faire dans une session dédiée) :**
- `front/static/front/js/keyboard.js` : retirer le raccourci `L` (toggle focus)
- `front/static/front/css/` : retirer les classes `.mode-focus`, `.mode-focus-active`
- Toolbar : retirer le bouton focus s'il existe
- Modale d'aide raccourcis : retirer la ligne `L`
- Tests E2E focus mode : suppression

**À surveiller :**
- Vérifier in vivo que Firefox Reader View s'active sur `/lire/{id}/` (l'icône doit
  apparaître dans la barre d'URL après chargement)
- Si Reader View ne se déclenche pas, ajouter un `<meta name="description">` et éventuellement
  une `<meta name="author">` pour aider l'algorithme
- Chrome n'a pas Reader View natif → c'est un compromis assumé, l'usage cible reste Firefox

## 3. Bibliothèque d'analyseurs (PHASE-26b, vue dédiée) — fusion dans menu config

**Décision :** retirer la vue "Bibliothèque d'analyseurs" en tant que vue dédiée. Garder
la gestion des analyseurs dans le menu de configuration LLM existant.

**Raison :** la vue dédiée duplique ce que le menu config offre déjà (sélection de modèle,
prompt, paramètres). Le besoin d'une grille de cartes admin avec compteurs de coûts ne
s'est pas matérialisé. L'admin édite ses analyseurs via une page DRF standard avec
permission `is_staff`.

**Impact code (à faire dans une session dédiée) :**
- `front/views_analyseurs.py` : conserver le ViewSet pour le CRUD admin via menu config, retirer les vues "vitrine"
- `front/templates/front/includes/analyseurs_*.html` : suppression des templates de la grille
- URLs : retirer la route `/analyseurs/` publique, garder uniquement `/config/llm/analyseurs/`
- `front/static/front/js/` : retirer le JS spécifique à la galerie
- Modale config LLM : intégrer la liste des analyseurs avec édition inline
- Tests : conserver les tests de cost tracking (utile), retirer les tests de la galerie

**Conservé :**
- Le modèle `Analyseur` (utilisé par tout le pipeline)
- L'historique des prompts (audit trail)
- Le calcul de coûts (utilisé pour les crédits Stripe PHASE-26h)
- L'édition admin-only via le menu config

## 4. Explorer (PHASE-25d / PHASE-25d-v2) — retrait complet

**Décision :** retirer la page Explorer (Découvrir / vitrine publique de dossiers). Ne pas
développer plus la dimension "réseau social" d'Hypostasia.

**Raison :** YAGNI. Hypostasia n'est pas un réseau social. Le partage entre utilisateurs
passe déjà par les invitations email (PHASE-25d, partie invitation conservée) et par les
groupes (PHASE-25c). La curation publique de débats controversés/sans-commentaire n'apporte
pas de valeur prouvée et oriente le produit vers une dynamique de "feed" qui n'est pas
voulue.

**Impact code (à faire dans une session dédiée) :**
- `front/views_explorer.py` : suppression complète
- `front/templates/front/includes/explorer_*.html` : suppression des 6 fichiers (`explorer_contenu`, `explorer_card_document`, `explorer_preview`, `explorer_curation`, `explorer_resultats`, `explorer_card`)
- URLs : retirer toutes les routes `/explorer/`
- `front/serializers.py` : retirer `ExplorerFiltresSerializer`
- `front/templates/front/includes/onboarding_vide.html` : retirer l'onglet "Explorer les dossiers", garder uniquement "Découvrir l'app"
- `front/templates/front/base.html` : retirer les liens HTMX explorer dans toolbar et sidebar
- `front/templates/front/includes/lecture_principale.html` : retirer `explorer_preloaded` cascade si présente
- Tests : retirer `Phase25dExplorerRechercheTest` et tests E2E associés

**Conservé :**
- L'invitation par email (partie utile de PHASE-25d)
- Les visibilités 3 niveaux (public/collaborateurs/privé) — la visibilité "public" reste
  utile pour partager via URL directe, sans nécessiter un Explorer
- Les groupes (PHASE-25c)

## Vérification Firefox Reader View — résultat

`front/templates/front/includes/lecture_principale.html` rend déjà :

```html
<h1 class="titre-page-cliquable text-2xl font-bold text-slate-900 mb-1 ...">
    {{ page.title|default:"Sans titre" }}
</h1>
...
<article id="readability-content" class="lecture-article prose prose-slate prose-lg max-w-none ...">
    {{ html_annote|default:page.html_readability|safe }}
</article>
```

Le rendu est compatible Mozilla Readability.js : titre `<h1>` unique, `<article>` avec
contenu prose long, paragraphes substantiels, peu de bruit dans le DOM critique.
La toolbar et les overlays sont en `position: fixed` et seront naturellement ignorés
par l'algorithme.

**À tester en vrai** : ouvrir une page `/lire/{id}/` dans Firefox et vérifier que
l'icône Reader View apparaît dans la barre d'URL. Si elle n'apparaît pas, ajouter
`<meta name="description">` dans `base.html` (utilise les premiers mots de
`page.html_readability`) pour booster le score Readability.
